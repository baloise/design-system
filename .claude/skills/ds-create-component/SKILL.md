# ds-create-component

Create a new Stencil component with complete scaffolding: component stub, TDD test files, visual regression tests, a11y tests, design tokens, and i18n support. Automatically mirrors backwards-compatible APIs from old `bal-` components when they exist.

## Quick start

```bash
/ds-create-component button
```

The skill will:

1. Scan the old `bal-button` component for API reference
2. Ask you interactive questions about component type and features
3. Generate stub files with failing tests (red phase TDD)
4. Wire up design tokens, i18n, and animation handling
5. Auto-export the component

## What it generates

```
packages/core/src/components/my-component/
├── my-component.tsx               # Component stub (implements interfaces)
├── my-component.interfaces.ts      # Props, events, enums (from old component if exists)
├── my-component.host.scss         # Shadow DOM styles (web component only)
├── my-component.style.scss        # Light DOM / global styles
├── my-component.i18n.ts           # Internationalization (if needed)
└── test/
    ├── my-component.component.play.ts   # Component interaction tests (generated by ds-sync-component-tests)
    ├── my-component.visual.play.ts      # Visual regression tests (generated by ds-sync-visual-tests)
    ├── my-component.visual.html         # Visual test harness
    └── my-component.a11y.play.ts        # A11y tests (generated by ds-sync-a11y-tests)
```

Plus:

- Updated `packages/core/src/index.ts` with component export
- Page Object in `packages/playwright/src/lib/components/my-component.po.ts`

## Generation Strategy

### Files created in order:

1. **interfaces.ts** — Props, events, enums (extracted from old component if exists)
2. **component.tsx** — Stub implementation (bare minimum to compile)
3. **host.scss / style.scss** — Empty SCSS with comments showing available tokens
4. **i18n.ts** — (if component uses aria-labels, titles, or text strings)
5. **Test files** — Via orchestrated skills:
   - `ds-sync-component-tests` → component interaction tests (red phase)
   - `ds-sync-visual-tests` → visual regression harness + tests
   - `ds-sync-a11y-tests` → accessibility tests
6. **Export** — Auto-added to `packages/core/src/index.ts`

### Test-Driven Development (red phase)

The component stub is **intentionally incomplete**. Generated tests will fail until you implement:

- Property validation (`validateProps()`)
- Event emissions
- Form integration (if form component)
- Accessibility attributes

This enforces red-green-refactor workflow.

## Backwards Compatibility Rules

If an old `bal-component` exists:

1. **Extract API** — Parse props, events, methods from old component
2. **Preserve where possible** — Use same prop names, event names, behavior
3. **Conflict resolution** — If old API conflicts with SEO/A11y best practices:
   - Mark the prop as `@deprecated` in JSDoc
   - Accept it but internally use modern approach
   - Document migration path in comments
4. **No prop aliases** — Don't create multiple names for same thing; just match the old name

### Example: Native button type

Old `bal-button` had: `native` prop (render `<button>` or `<div role="button">`)

New `ds-button`:

```tsx
/**
 * @deprecated Use semantic HTML button elements. This prop is accepted for API
 * compatibility but is ignored. Always renders <button> for better a11y.
 *
 * @see https://[your-migration-docs]
 */
@Prop()
readonly native?: boolean // Accepted but unused
```

## Component Types

### 🔹 wc-only (Web Component)

Shadow DOM only. No global styling.

**Files:**

- `component.tsx`
- `component.host.scss` (shadow DOM styles)

**Use when:** Styling is fully isolated, no light DOM customization needed.

### 🔹 hybrid

Both shadow DOM + light DOM support.

**Files:**

- `component.tsx`
- `component.host.scss` (shadow DOM)
- `component.style.scss` (light DOM / global styles)

**Use when:** Component needs both isolated internal styling + customizable light DOM layout.

### 🔹 css-html

Pure HTML/CSS utility, no web component.

**Files:**

- `component.style.scss` only
- No `component.tsx`

**Use when:** Simple presentational component (e.g., grid helper, spacing utility).

## Form Component Requirements

If the component is a form component (form input, select, checkbox, etc.):

**Must have:**

- `name: string` prop
- `value: T` prop (generic, e.g., string, boolean, number)
- `disabled: boolean` prop
- `required: boolean` prop (if applicable)
- `invalid: boolean` prop (if validation needed)
- `@AttachInternals() internals!: ElementInternals`
- `formAssociated: true` in `@Component()` decorator
- `validateProps()` method called from `connectedCallback()` and `componentWillUpdate()`
- Form submission/reset tests

**Auto-wired by skill:**

```tsx
@AttachInternals() internals!: ElementInternals

@Watch('value')
valueChanged(newValue: T) {
  this.internals.setFormValue(newValue)
}
```

## Design Tokens Integration

The skill scans `packages/tokens/tokens/Base.tokens.json` and:

1. Extracts available color, typography, spacing tokens
2. Shows available tokens in SCSS as comments:
   ```scss
   // Available color tokens:
   // --ds-color-primary-{1..10}
   // --ds-color-danger-{1..10}
   // --ds-color-text
   ```
3. Warns if old component references missing tokens
4. Auto-generates CSS variable cascade (private → public → modifier → token)

## i18n (Internationalization)

If the component has aria-labels, titles, or user-facing text:

1. **Extracts English strings** from old component (if exists)
2. **Generates i18n.ts** with structure:
   ```ts
   export const i18nDsMyComponent: I18n<I18nDsMyComponent> = {
     de: { close: 'Schliessen', ... },
     en: { close: 'Close', ... },
     fr: { close: 'Fermer', ... },
     // ... other languages
   }
   ```
3. **Auto-wires listener** if i18n is needed:
   - Implements `DsConfigObserver`
   - Uses `@ListenToConfig()` decorator
   - Reads current language and applies strings

## Animation Handling

If the component uses CSS animations:

1. **Auto-detects** animation/transition properties in SCSS
2. **Wires config listener** to respect `dsConfig.animated` flag
3. **Generates no-animation class:**

   ```tsx
   <Host class={{ 'no-animation': !this.animated }}>

   :host(.no-animation) * {
     animation: none !important;
     transition: none !important;
   }
   ```

Visual regression tests automatically disable animations via dsConfig.

## Browser Support

Components must support **latest 2 major versions of each browser** (Chrome, Firefox, Safari, Edge).

The skill validates that APIs used are baseline available on all modern browsers:

- ✅ ElementInternals (form components)
- ✅ Shadow DOM / CSS custom properties
- ✅ CSS Grid / Flexbox
- ❌ APIs marked as experimental on caniuse.com
