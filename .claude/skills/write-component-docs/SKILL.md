# Write Component Docs

Generates a complete documentation set for a component in `docs/src/components/<component>/`. The canonical reference for structure and style is the **accordion** component (`docs/src/components/accordion/`).

Each component gets exactly **five MDX files** (no SVG parts or static tables — now dynamically rendered):

| File                  | Purpose                                                           |
| --------------------- | ----------------------------------------------------------------- |
| `1-Overview.mdx`      | Live canvas + playground + ComponentLead + ComponentPublicMethods |
| `2-Usage.mdx`         | When to use, do's/don'ts, `UsageExamples`                         |
| `3-Variants.mdx`      | All story variants with `Canvas`                                  |
| `4-Styling.mdx`       | Shadow DOM parts (ComponentParts) + CSS variables                 |
| `5-Accessibility.mdx` | WCAG guidelines via `A11yGuidelines`                              |

## Supporting Files (Auto-generated)

| File                        | Purpose                              | Meta Title Pattern                  |
| --------------------------- | ------------------------------------ | ----------------------------------- |
| `<component>.stories.ts`    | Stencil story exports (all variants) | `Components/<Category>/<Component>` |
| `<component>.doc-config.ts` | Shared section/color/tabs config     | N/A (TypeScript config)             |

**Important:** The `<component>.stories.ts` file should have **only the component name** in the `title` field, **NOT** `/Variants` or any subpath. The MDX files (1-Overview, 2-Usage, 3-Variants, 4-Styling, 5-Accessibility) each define their own full path in their Meta tags.

```ts
// ✅ Correct: stories.ts
const meta: Meta<Args> = {
  title: 'Components/Feedback/Toast', // Just the component, no /Variants
  // ...
}
```

```mdx
// ✅ Correct: 3-Variants.mdx

<Meta title="Components/Feedback/Toast/Variants" />
```

Do NOT create `api.md` (auto-generated by Stencil), `.parts.svg`, or any other files.

## New Reusable Storybook Blocks

### ComponentLead

Displays the component description (from `components.json` `overview` or `docs` field) with the first word bolded if it matches the component name.

```mdx
import { ComponentLead } from '../../../.storybook/blocks'

<ComponentLead component="accordion" />
```

**Output:** `<strong>Accordion</strong> displays collapsible content sections with open/close toggle buttons...`

### ComponentPublicMethods

Displays all public methods from `components.json` with signatures, parameters, return types, and documentation.

```mdx
import { ComponentPublicMethods } from '../../../.storybook/blocks'

<ComponentPublicMethods
  component="alert-container"
  title="Available Methods"
  description="These methods allow you to programmatically control the component from your application code."
/>
```

**Props:**

- `component` (required): component tag or name
- `title` (optional): heading for the methods section
- `description` (optional): introductory text

Shows a friendly message if no methods are available.

### ComponentParts

Displays all Shadow DOM parts from `components.json` in a styled table.

```mdx
import { ComponentParts } from '../../../.storybook/blocks'

<ComponentParts component="accordion" />
```

**Output:** Auto-generated table with Part name and Description columns from JSDoc `@part` tags.

## Process

### Step 1 — Gather context

Read these files to understand the component:

- `packages/core/src/components/<component>/<component>.tsx` — props, events, parts, render output
- `packages/core/src/components/<component>/test/<component>.visual.html` — story sections
- `packages/core/src/components/<component>/test/<component>.style.html` — CSS-only sections (if exists)

### Step 2 — Determine category and color

Ask the user if not obvious:

- **Category**: the Storybook sidebar path after `Components/` (e.g. `Data Display`, `Navigation`, `Forms`)
- **Brand color**: the accent color for the Banner header — pick one of `red`, `blue`, `green`, `yellow`, `purple` that fits the component's purpose

The category determines the Storybook `title` and the `doc-config` section string.

### Step 3 — Present story sections

Show ALL `data-testid` sections from `visual.html` as a numbered list:

```
Which sections should become story variants?

[ ] 1. basic
[ ] 2. with-icon
[ ] 3. colors
[ ] 4. sizes
...

(enter numbers separated by commas, or "all")
```

Wait for user selection before generating anything.

  component="<component>"

```

**Key changes:**

- `<ComponentLead component="<component>" />` — replaces manual `<Lead>` JSX; pulls from `components.json`
- `<ComponentPublicMethods />` — replaces empty space; shows public methods if any exist

## When to Use

- [bullet list of valid use cases for this component]

## When NOT to Use

- [bullet list of anti-patterns]

<UsageExamples

## File 6 — `3-Variants.mdx`

One `Canvas` block per selected story variant, each preceded by a `StoryHeading` and a short description.

```mdx
import { Canvas, Markdown, Meta } from '@storybook/addon-docs/blocks'
import {
  Banner,
  BannerTabs,
  Footer,
  StoryHeading,
} from '../../../.storybook/blocks'
import * as <ComponentName>Stories from './<component>.stories'
import { <COMPONENT>_DOC_CONFIG, get<Component>Tabs } from './<component>.doc-config'

<Meta title="Components/<Category>/<Name>/Variants" />
<StoryHeading of={<ComponentName>Stories.Basic} hidden></StoryHeading>
<Banner label={'Variants'} section={<COMPONENT>_DOC_CONFIG.section} color={<COMPONENT>_DOC_CONFIG.color} />
<BannerTabs
  of={<ComponentName>Stories}
  tabs={get<Component>Tabs('variants')}
/>

{/* STORIES */}
{/* ------------------------------------------------------ */}

<StoryHeading of={<ComponentName>Stories.<StoryName>}></StoryHeading>

[One or two sentence description of this variant.]

<Canvas of={<ComponentName>Stories.<StoryName>} />

{/* ------------------------------------------------------ */}

<StoryHeading of={<ComponentName>Stories.<NextStoryName>}></StoryHeading>

[Description.]

<Canvas of={<ComponentName>Stories.<NextStoryName>} />

<Footer />
```

Add `{/* ------------------------------------------------------ */}` before each `StoryHeading`. End the page with `<Footer />`.

## Parts

Shadow DOM parts are named elements within a web component that can be targeted with CSS selectors from outside the component, allowing you to style specific internal elements without breaking encapsulation.

<br />

**Example:** Target specific parts to customize appearance:

```css
ds-<component > ::part(<part-name >) {
  /* your overrides */
}
```
````

<br />

Primary parts of the <component> component include:

<ComponentParts component="<component>" />

## Design Tokens

CSS variables (custom properties) allow you to define reusable values that can be customized globally or per component instance to override default styling without modifying the component code.

<br />

**Example:** Override styling using CSS variables:

```css
:root {
  --ds-<component>-color-<variant>-base-text: #ffffff;
  --ds-<component>-color-<variant>-base-background: #188d5b;
}
```

<TokenOverview component="<component>" />

<Footer />
```

**Key changes:**

- Removed static SVG import
- Removed hardcoded markdown table
- Use `<ComponentParts component="<component>" />` — pulls parts from `components.json`

<br />

**Example:** Override <component> color and styling using CSS variables:

## File 8 — `5-Accessibility.mdx`

WCAG guidelines rendered via the `A11yGuidelines` block. Content must be **consumer-facing** — what the component _user_ must do, not what the component does internally.

```mdx
import { Canvas, Markdown, Meta } from '@storybook/addon-docs/blocks'
import {
  A11yGuidelines,
  Banner,
  BannerTabs,
  Footer,
} from '../../../.storybook/blocks'
import * as <ComponentName>Stories from './<component>.stories'
import { <COMPONENT>_DOC_CONFIG, get<Component>Tabs } from './<component>.doc-config'

<Meta title="Components/<Category>/<Name>/Accessibility" />
<Banner label={'Accessibility'} section={<COMPONENT>_DOC_CONFIG.section} color={<COMPONENT>_DOC_CONFIG.color} />
<BannerTabs
  of={<ComponentName>Stories}
  tabs={get<Component>Tabs('accessibility')}
/>

## Guidelines

<A11yGuidelines
  items={[
    {
      type: 'do',
      title: '[Guideline Title]',
      content: <p style={{ margin: '0.5rem 0 0 0' }}>[Actionable instruction for the component consumer.]</p>,
    },
    {
      type: 'dont',
      title: '[Anti-pattern Title]',
      content: <p style={{ margin: '0.5rem 0 0 0' }}>[What to avoid and why.]</p>,
    },
  ]}
/>

## References

- [MDN: aria-label](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-label) — [one-line purpose]
- [MDN: aria-hidden](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-hidden) — [one-line purpose]

<Footer />
```

`A11yGuidelines` accepts `items` of `{ type: 'do' | 'dont', title: string, content: ReactNode }`.

### Accessibility content principle

Document **consumer requirements only**. Do not describe internal implementation details.

```
✓ Add an aria-label to the parent element when the badge shows only a number.
✗ The component renders a <span> with part="badge" internally.
```

## Shared Patterns

### BannerTabs constraint

`Meta title` must always be a **string literal** — never a template literal or computed value. Storybook's static indexer cannot evaluate expressions.

```mdx
{/* ✓ correct */}

<Meta title="Components/Data Display/Badge/Usage" />

{/* ✗ wrong — causes indexing error */}

<Meta title={`${BADGE_DOC_CONFIG.section}/Usage`} />
```

### Import only what you use

Each MDX page should import only the blocks it actually renders. Do not copy the full import list from another page.

### Section dividers

Use `{/* ------------------------------------------------------ */}` before every `StoryHeading` in `3-Variants.mdx`.

### Key story utilities

- `withDefaultContent(text?)` — initial slot text arg
- `withContent()` / `withComponentControls({ tag })` — argTypes
- `withRender(fn)` — render function
- `props(args)` — serialises Stencil props to HTML attribute string
- `StoryFactory<Args>(meta)` — returns the `Story()` helper

## Reference: Accordion Component

The accordion is now the canonical example for the new block-based pattern. When in doubt, mirror its file structure:

```
docs/src/components/accordion/
  accordion.stories.ts       ← stories, meta title ends in /Variants
  accordion.doc-config.ts    ← ACCORDION_DOC_CONFIG, getAccordionTabs()
  1-Overview.mdx             ← ComponentLead + Canvas + ComponentPublicMethods
  2-Usage.mdx                ← UsageExamples block
  3-Variants.mdx             ← StoryHeading + Canvas per variant
  4-Styling.mdx              ← ComponentParts + TokenOverview
  5-Accessibility.mdx        ← A11yGuidelines + References
  api.md                     ← AUTO-GENERATED — never edit
```

Note: No `.parts.svg` or static content tables — these are auto-generated from `components.json` and Stencil JSDoc.
