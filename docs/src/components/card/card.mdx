## Accessibility

### HTML / CSS only

Use the semantic `<article>` element to contain your card. Provide a proper heading hierarchy, use semantic elements for structure, and avoid relying on color alone to communicate meaning.

<Markdown>
  {`
| Requirement         | How to implement                                                                                       |
| ------------------- | -------------------------------------- --|
| Card container      | Use \`<article>\` to wrap the card; it signals to screen readers that this is self-contained content   |
| Heading hierarchy   | Always include a heading (\`<h2>\`, \`<h3>\`, etc.) as the card's title; respect overall page hierarchy |
| Semantic structure  | Use \`<header>\` for the title area, \`<div>\` or \`<section>\` for content, \`<footer>\` for actions  |
| Color not sole cue  | Do not rely on card color (via \`is-primary\`, \`is-danger\`, etc.) to communicate meaning alone       |
| Clickable cards     | If the entire card is clickable, add \`role="button"\` and \`tabindex="0"\`; handle Enter/Space keys  |
| Close buttons       | Use \`<ds-close>\` for closable cards; it handles keyboard activation automatically                    |
| Interactive content | Ensure any form controls, buttons, or links within the card are properly labeled                       |
`}
</Markdown>

> **Note:** The card component is a **container only**. Accessibility depends entirely on the semantic HTML and interactive elements you place inside it. Always provide a descriptive heading as the first child of the card header.

### Web component

The `ds-card` element provides a semantic wrapper and slot management:

- **`<article>` wrapper** — renders the main card container as an `<article>` element in the shadow DOM
- **Slot projection** — allows you to place any semantic HTML inside; your heading and content structure is preserved
- **No built-in interactivity** — the component is a layout container; if `clickable={true}`, af
- **Color semantics** — the `color` prop affects visual styling only; it does not set any ARIA roles or affect meaning
- **Shadow DOM encapsulation** — styles are scoped; nested interactive elements (buttons, links, form controls) remain fully accessible
