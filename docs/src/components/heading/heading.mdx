## Accessibility

### HTML / CSS only

Use native heading elements so assistive technologies can build a correct document outline.

<Markdown>
  {`
| Requirement              | How to implement                                                                                                             |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| Semantic heading level   | Choose the element (\`<h1>\`–\`<h6>\`) that reflects the correct position in the document outline, not the visual size      |
| Visual level override    | Use \`class="heading is-{level}"\` (e.g. \`is-3\`) to change visual size without affecting the document outline            |
| Unique \`<h1>\`          | Use only one \`<h1>\` per page to identify the main topic                                                                    |
| Sequential heading order | Do not skip heading levels (e.g. jump from \`<h2>\` directly to \`<h4>\`) — this breaks screen reader navigation           |
| Meaningful text          | Heading text must describe the section it introduces; avoid generic labels like "Section" or "Title"                        |
`}
</Markdown>

### Web component

The `ds-heading` element manages the following automatically:

- **Semantic tag** — renders the correct `<h1>`–`<h6>` element based on the `level` prop; the tag in the DOM always reflects the semantic level, not the visual style
- **Visual decoupling** — `visual-level` changes the CSS size class without altering the rendered heading tag, allowing visual hierarchy to differ from document hierarchy without introducing accessibility violations
- **Auto-level** — `auto-level` dynamically shrinks the visual size when content wraps; the semantic tag is never changed

> **Note:** The component does not enforce document outline correctness — it is the developer's responsibility to choose the right `level` value for the page context.
