## Introduction

Helvetia Design System is free to use for anybody building a Helvetia product or website, and the Helvetia community is always working to make it better.
Contributors like you help to make Helvetia Design System great, and so we’re glad you’re here.

Contributions are not limited to code. We also encourage feedback, documentation, new designs, and tools.

All you need is a [public GitHub account](https://github.com/) to get started. Most contributions begin with a GitHub issue using one of these templates:

## Community

### Users

Users are members of the community who use Helvetia Design System guidelines, assets, and tooling. Anyone can be a user, and we encourage users to participate in the community as much as possible.

### Contributors

Contributors are members of the community who contribute to Helvetia Design System in a material way. Anyone can be a contributor. In addition to participating as a user, you can also contribute by:

- Reporting bugs or missing features through GitHub issues
- Fixing bugs, adding features, and improving documentation

### Maintainers

Maintainers are members of the community who are committed to the success of individual Helvetia Design System projects. In addition to their participation as a contributor, maintainers:

- Label, close, and manage GitHub issues
- Close and merge GitHub pull requests

## Workflow

The Helvetia Design System typically uses a [fork and pull request workflow](https://docs.github.com/en/get-started/quickstart/contributing-to-projects) for contributions.

### Quick Start for Contributors

**New to contributing?** Here's the typical flow:

1. **Start with an issue** — Create or find an issue to work on (see buttons above)
2. **Fork & branch** — Fork the repo, create a branch (e.g., `feat/button-variant`)
3. **Make changes** — Follow the contributing guidelines and code standards
4. **Test & commit** — Write tests, run linting, create changesets
5. **Open a PR** — Push your branch and open a pull request against `next`
6. **Address feedback** — Core team reviews and suggests improvements
7. **Merge & release** — Once approved, your changes are merged and published

See [CONTRIBUTING.md](https://github.com/baloise/design-system/blob/next/CONTRIBUTING.md) for detailed step-by-step instructions.

## Pro Tips for Contributors

### Test Your Changes with Snapshots

Before merging, publish a **snapshot version** to test in a real project:

```bash
# Post this comment in your PR:
/snapshot

# Then install and test:
npm install @baloise/ds-core@1.2.3-snapshot.456.abc1234
```

See [ARCHITECTURE.md — Snapshot Versions](https://github.com/baloise/design-system/blob/next/ARCHITECTURE.md#snapshot-versions-testing-before-merge) for details.

### Use Bot Commands

Post these as PR comments to trigger automation:

- **`/update-screenshots`** — Regenerate visual regression snapshots after intentional changes
- **`/snapshot`** — Publish a pre-release version for testing
- **`/cib`** — Create a branch from an issue (post on the issue, not the PR)

See [CONTRIBUTING.md — Bot Commands](https://github.com/baloise/design-system/blob/next/CONTRIBUTING.md#bot-commands) for more.

### Understand Semantic Versioning

When creating a changeset, choose the right version bump:

- **Patch** — bug fixes, non-breaking improvements
- **Minor** — new features, new components
- **Major** — breaking changes (prop renames, removed props, behavior changes)

See [ARCHITECTURE.md — Semantic Versioning Policy](https://github.com/baloise/design-system/blob/next/ARCHITECTURE.md#semantic-versioning-policy) for detailed guidance with examples.

## Design Feedback & Ideas First?

**Not sure if your idea is good?** That's okay!

1. **Open a discussion issue** — Use the 💬 "Ask a question" button to discuss your idea first
2. **Get feedback early** — The core team can provide guidance before you spend time implementing
3. **Iterate together** — Work through design/technical considerations with the team
4. **Then implement** — Once approved, you're ready to code with confidence

This saves time and ensures your contribution aligns with the design system's vision.
