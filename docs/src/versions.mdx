#### Understanding Version Numbers

The Helvetia Design System uses **Semantic Versioning** (MAJOR.MINOR.PATCH):

<Markdown>
  {`
| Type | Format | When | Examples |
|------|--------|------|----------|
| **Patch** | X.X.↑ | Bug fixes, accessibility improvements, non-breaking tweaks | 18.0.0 → 18.0.1 |
| **Minor** | X.↑.0 | New features, new components, new props (backward-compatible) | 18.0.0 → 18.1.0 |
| **Major** | ↑.0.0 | Breaking changes: removed props, renamed events, API changes | 17.0.0 → 18.0.0 |
`}
</Markdown>

**Why this matters:**

- A `patch` update is always safe.
- A `minor` update adds features safely.
- A `major` update requires attention and testing.

<br />

#### Version Support Policy

<Markdown>
  {`
| Version | Status | Support | Security Fixes |
|---------|--------|---------|-----------------|
| **Latest major** (e.g., 19.x) | Supported | ✅ Full support | ✅ Yes |
| **Older majors** (e.g., 18.x, 17.x) | EOL | ⚠️ Maintenance only | ❌ No |
`}
</Markdown>

**What this means:** Only the latest major version receives security patches and bug fixes. Upgrade regularly to stay secure.

<br />

#### Breaking Changes

All **breaking changes** are documented and require a major version bump. Common breaking changes:

- ❌ Prop removal or rename
- ❌ Event removal or rename
- ❌ Method signature changes
- ❌ Component removal
- ❌ Default value changes that affect behavior
- ❌ Shadow DOM slot or part changes

When a breaking change is made, the upgrade guide explains the migration path.
